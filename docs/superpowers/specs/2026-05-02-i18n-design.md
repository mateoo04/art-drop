# i18n Design

**Date:** 2026-05-02
**Scope:** Frontend only ([artdropapp-frontend/](../../../artdropapp-frontend/)).
**Languages:** English (default, fully populated) + Croatian (scaffolded, empty translation file).

## Goals
- Replace every user-facing string in the React frontend with translation keys.
- Detect the user's language from the browser on first visit; persist their choice to `localStorage` after they switch.
- Expose a language switcher inside a new **Settings** drawer opened from the gear icon next to the edit pencil on `/account`.
- Ship Croatian as an empty stub so the user can fill it in later without code changes; missing keys fall back to English.

## Non-goals
- No backend translation (Spring Boot API messages remain English).
- No Croatian translations authored by Claude — only the empty file.
- No locale-aware date / number / currency formatting beyond `Intl` defaults.
- No syncing of locale preference to the user's DB record.

## Library choice
- `i18next` + `react-i18next` + `i18next-browser-languagedetector`.
- Reasoning: standard React stack, namespace support for future growth, built-in detector with `localStorage` caching, graceful fallback for missing keys.

## Architecture

### Initialization
- New file `src/lib/i18n.ts` configures i18next:
  - Resources for `en` and `hr`, namespace `common`.
  - `LanguageDetector` order: `localStorage` → `navigator` → fallback `en`.
  - `localStorage` key: `artdrop.lang`.
  - `fallbackLng: 'en'`, `returnEmptyString: false` so empty HR keys fall through to EN.
- `src/main.tsx` imports `./lib/i18n` once at module load (side-effect import) before `<App />` renders.

### Translation files
- `src/locales/en/common.json` — populated.
- `src/locales/hr/common.json` — `{}` (empty stub).
- Key structure (flat-ish, grouped by area):
  - `nav.*` — header/footer/nav items
  - `auth.*` — login, signup, validation
  - `home.*` — homepage copy, empty states
  - `profile.*` — profile header, circle button, edit sidebar
  - `account.*` — account page, settings drawer, seller section
  - `artwork.*` — detail, drop, edit, comments
  - `challenges.*` — list, detail, past list
  - `collections.*`, `circle.*`, `admin.*`
  - `common.*` — shared (Save, Cancel, Loading, error messages)

### Usage pattern
- Components call `const { t } = useTranslation()` and replace literal strings with `t('section.key')`.
- Interpolation for dynamic values: `t('profile.followCta', { name: profile.displayName })`.
- Zod validation messages are keyed via `t(...)` inside `useMemo`-built schemas (so they re-translate on language change).

### Settings drawer
- New component `src/components/account/SettingsSidebar.tsx`:
  - Mirrors `ProfileEditSidebar` patterns: backdrop, right-side `aside`, slide-in transition, Esc-to-close, `aria-modal`.
  - Props: `{ open: boolean, onClose: () => void }`.
  - Header: "Settings" title + close `X`.
  - Body: a single labeled `Language` row containing a `<select>` with options `English` and `Hrvatski`. On change calls `i18n.changeLanguage(value)`.
  - Designed for future rows (theme, notifications) — uses a vertical-stack layout under the header, not a form.

### AccountPage wiring
In `src/pages/AccountPage.tsx`, the `action` slot of `ProfileHeader` becomes:
```
[ Settings gear icon button ]  [ Edit pencil icon button ]
```
- New `settingsOpen` state alongside the existing `editOpen`.
- New `<SettingsSidebar open={settingsOpen} onClose={() => setSettingsOpen(false)} />` rendered as sibling to `<ProfileEditSidebar />`.
- Gear: `Settings` icon from `lucide-react`, same className as the pencil button.

## String migration scope
Every `.tsx` file containing user-facing copy. Buckets:

- **Layout / nav** — `AppHeader`, `AppFooter`, `AuthHeader`, `NewDropFab`, `MainLayout`
- **Auth** — `LoginPage`, `SignupPage`, `AuthPromptContext`
- **Home / feed** — `HomePage`, `MasonryFeed` empty/loading states
- **Profile / Account** — `ProfilePage`, `AccountPage`, `ProfileHeader`, `ProfileEditSidebar`, `SellerSection`, `SellerStatusBadge`, `SellerApplicationModal`
- **Artwork** — `ArtworkDetailPage`, `ArtworkDetailComponent`, `ArtworkDropPage`, `ArtworkEditPage` and their child components
- **Challenges** — `ChallengesPage`, `ChallengeDetailPage`, `PastChallengesList`, et al.
- **Circle, Collections** pages
- **Admin** — `AdminLayoutPage`, `AdminUsersPage`, `AdminUserDetailPage`, `AdminRoute` denial copy
- **Shared UI** — `Spinner` labels, `Button` loading text, `FormField` error rendering, `ProtectedRoute` messages
- **Toasts** (`sonner`) — success / error messages from API hooks
- **Validation** — Zod schemas with literal messages

API-returned error strings (e.g. server validation) remain English in this pass.

## Data flow
1. App boots → `i18n.ts` runs; detector reads `localStorage[artdrop.lang]`, then `navigator.language`, picks `en` or `hr`.
2. Components call `useTranslation()` and re-render when language changes.
3. User opens Settings drawer → picks language in `<select>` → `i18n.changeLanguage(value)` → detector caches it to `localStorage` → all `t()` calls re-render.
4. HR is selected but most keys missing → `fallbackLng: 'en'` returns the English string.

## Testing
- Manual: switch to HR in the drawer, verify the few HR keys (none initially) fall back to EN cleanly with no `undefined`/empty UI.
- Type-check: `tsc -b` passes.
- Build: `npm run build` succeeds.
- Spot-check each major page in the browser after migration.

## Risks / notes
- 85 files is a lot of edits. The migration will be done page-area by page-area in the implementation plan, with a build/typecheck after each major bucket.
- `react-hook-form` + Zod schemas defined at module top-level need to be moved inside components (or built via a factory that takes `t`) so messages translate.
- `lucide-react` `Settings` icon already implicitly available (the package is a dependency).
